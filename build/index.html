About Us

With significant increases each year, cyber attacks are already a recurrent and worrisome topic that is expected to become even worse with the growing phenomenon of IoT (Internet of Things) and the millions of new interconnected devices that will enter the world wide web. At LabSecIoT, we aim to study information security problems in these new digital infrastructures, particularly in the domain of Smart Cities, where most systems have been found lacking the necessary protection against cyberwarfare, and consequently devise, with the proper development and testing, possible solutions for said issues considering the usage of adequate security communication protocols, the implementation of efficient Access Control techniques and the integration of digital certification.

Current Challenges

  • Create a pedagogical framework to promote the innovation underlying the IoT concept development and, in particular, stimulate the security-by-design paradigm.
  • Develop conceptual and prototyping work for a "true smart city" based on an horizontal architecture and an all-inclusive philosophy underlined by high integration and high performance standards.
  • Create a virtual environment that is capable of integrating high numbers of small devices with different communication capabilities.
  • Design devices to work as sensors, being capable of acquiring information from the environment, such as humidity, temperature, light, etc...

edge4all

Innovative edge-based architecture for secure and integrated Smart Cities

This project aims to justify an Edge Computing based architecture for a complex scenario such as a Smart City. A Smart City can be described (from a technical point of view) as thousands or even millions of small disposable devices (IoTs), each one with a specific goal and endowed with wireless communication capabilities. Since the Cloud has bandwidth limitations mainly concerning number of connections, it becomes evident the need to introduce an intermediate stage - the Edge - between the devices and the Cloud. This Edge is oriented to receive as many low-level data (logs) as possible, process them in real time generating high-level information, and store it in databases for latter analysis. The purpose of such architecture is to feed the Cloud only with high-level information (instead of millions of minimal logs), where some machine-learning and big-data algorithms may be implemented. Concerning to the Edge itself, it is: capable of validate IoTs security (concerning integrity and confidentiality); solid and robust enough to handle a huge number of logs/connections; efficiently transform low-level data into high-level information; offer a proper API to integrate with higher-level systems; and easily scalable.

The edge4all project is is specifically oriented to Smart City scenarios, intended to work as an information provider for the common citizen, a decision-making help desk for governments and business, and a device management system for the devices' owners. Features as history, filters and warnings are just some of the already implemented features.

Smart City Simulator

High performance virtual environment for testing smart city platforms.

In order to demo the edge4all project, the team designed a virtual environment that simulates a real smart city, represented by large quantities of different devices (in mobility, services they provide, security level, etc...). This simulator is comprised of:

(a) a virtual city, complete with a proper coordinate system and roads;
(b) five personalized, independent, interactive and time-bound services (temperature, noise, air quality, UV Index and water drainage levels) from which devices gather data;
(c) a device management interface, allowing users to add, manipulate and test their own public or secure device in the virtual environment.

Disposable IoT Kits

Geo-referenced, wifi/gsm, low-powered device designed as a generic, low-cost IoT.

As the variability and availability of sensors increases, a need arose for a generic black-box kit that would be compatible with most of all these sensors. Features such as geo-referencing and Internet communications are often seen as requirements for many applications and as such mandatory attributes of any truly modern device. Our proposal for a generic IoT aims to fulfill all these requirements by delivering those features, allowing for communication via gsm or wifi; providing geo-referencing capabilities via gps and providing three inputs (digital or analog) so the user can connect any sensor. Due to very oriented developed libraries, this device, simply carrying an arduino pro mini, makes available about 1.5KB of memory for the user to interact with their sensors, enabling the usage of a private key. All of this can be supplied with 5 to 24 V with everything packed in a custom waterproof case, giving the kit a truly small footprint.

SCLP: Smart City Lightest Protocol

Lightweight protocol oriented to the security and privacy of IoT devices in Smart Cities

Despite current attempts for achieving lightweight and secure protocols, such as MQTT and CoAP, it is considered that their main aim has not been achieved yet. If a small device intends to send an authenticated message every 30 seconds over MQTT using a x509 certificate, it will use approximately 35MB of data per month*. As this amount of data is currently too expensive with telecommunication operators, specially considering a monthly fee per IoT, this project intends to lower the data consumption to 1MB per month to avoid fees.

To achieve this particular goal, the Smart City Lightest Protocol (SCLP) was developed as a very usecase oriented protocol that adds an extra layer of security to the whole system: the "device fingerprint". This protocol carries a tcp server where the logic is executed and a Redis key-value map where device information is stored. To interact with it, the device must previously register itself with the platform where some parameters are defined (like authentication type, communication frequency, what information will be sent,...). With this setup, the server can "define" the device "personality" (or deviceonality) that will be confirmed every message. If desired upon registration, the server can track the device by status (online of offline), by position (the device location at the last message) and/or by data, keeping an history of the information sent by the device.

Considering the above setup with MQTT (communication with certificate every 30 seconds), with every feature enabled, this new protocol uses 1.08MB per month.

*For this calculation, it is considered that the communication is established every message, because the devices "goes to sleep" to save energy.

Smart Mobility

To analyze, predict and optimize vehicle traffic

This project aims at improving the traffic wherever possible, starting with the cities of Guimarães (ptp. Vitrus) and Braga (ptp. TUB). We are training models with machine learning algorithms that will predict the future traffic so it is possible to improve it and optimize public transportation routes, reducing waiting times. We are also analyzing the data that it's possible to collect in order to analyze it to find patterns and key locations where the traffic is worse than average making it possible to act accordingly in order to make the most of everyone's commuting time.
